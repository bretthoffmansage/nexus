# Nexus Vault Expansion Pass Calendar Handoff v1

Status: implemented — 2026-07-02
Contract: `nexus_vault_expansion_pass_calendar_handoff_v1`
Nexus counterpart: `docs/specs/nexus_vault_expansion_pass_tool_v1.md`

## Summary

Adds the no-input, write-capable **Vault Expansion Pass** governed tool to the
Nexus Calendar and Skills page, reusing the exact Membership.io full-sync
scheduled-tool pattern. Nexus only schedules and enqueues; the existing Nexus
Connector claims and hands off; Nexus owns all execution. No new page, route,
queue, worker, or Connector path is introduced.

## Reused Membership full-sync path (not duplicated)

- `convex/lib/calendarScheduledTools.ts`: a new `VAULT_EXPANSION_PASS_SCHEDULED_TOOL`
  (`inputMode: "no_input_action"`, `writeCapable: true`, `chatAvailable: false`,
  `requiresConnectorCapability: true`, `singleFlightKey`,
  `executionTimeoutSeconds`) added to `CALENDAR_SCHEDULED_TOOLS`, plus
  `buildVaultExpansionPassTaskMetadata` — the same shape/idempotency as
  `buildMembershipFullSyncTaskMetadata`.
- `convex/scheduledEventDispatch.ts`: a `vault_expansion_pass` branch mirrors the
  membership branch to materialize the task on the existing `nexusTasks` queue
  via the existing scheduled-event maintenance/dispatch process.
- `convex/lib/p6config.ts`: `VAULT_EXPANSION_PASS_TOOL_ID` added to
  `KNOWN_CONNECTOR_TOOL_IDS` (operator must explicitly allow it) and classified
  `write_requires_confirmation`; intentionally NOT on `DEFAULT_CONNECTOR_TOOL_IDS`.
- `convex/schema.ts`: `vault_expansion_pass` task kind + a metadata validator.

## Canonical envelope (no-input)

```json
{
  "requestedToolId": "vault.expansion_pass",
  "taskKind": "vault_expansion_pass",
  "requestText": "Run Vault Expansion Pass",
  "taskMetadata": {
    "kind": "vault_expansion_pass",
    "sourcePage": "nexus_calendar",
    "explicitUserAction": "run",
    "scheduledEventId": "<event id>",
    "scheduledForUtc": "<ISO UTC instant>",
    "idempotencyKey": "<scheduledEventId>:<scheduledForUtc>"
  }
}
```

`requestText` is the fixed Nexus-side canonical string (generated by Nexus from
the tool definition, never user free text). The Connector strips/rejects any
model, provider, worker, prompt, path, timeout, attachment, or arbitrary
metadata field. The prompt is vault-owned and never appears in Nexus.

## Calendar integration

Selectable Calendar tool with a clear display name and a write-capable warning
description ("The pass may create and update Markdown notes across the active
Vault"). No task-request/prompt/model/path/advanced inputs (the
no-input dialog path shared with Membership full sync). Scheduled once; recurrence
only via the existing Calendar recurrence framework. Availability is gated on an
explicit Connector allowlist entry (shows unavailable otherwise). Uses the
existing queue, scheduled-task materialization, single-flight, and idempotency —
no duplicate runs on refresh/reconnect, no automatic run on event creation.

## Skills page

`convex/lib/nexusSkillsCatalog.ts`: a Scheduled Maintenance card — name, short
description, write-capable/Calendar availability, Connector status via the
existing pattern. Access modes are `["calendar", "connector"]` only: **no direct
Run button, no second execution path.** The full prompt is never shown in Nexus.

## Scope guarantees

No direct Nexus→Claude, Nexus→Codex, or Nexus→vault path; no new Nexus→Nexus
endpoint; no second queue, worker, daemon, scheduler, dispatcher, callback, MCP
server, or agent runtime. The existing Connector remains the sole handoff and
result path. The prompt/vault-root/worker/model/timeout/protected-paths are all
Nexus-owned.

## Live activation

Operator: allow the Connector capability
(`connectorRegistry:setConnectorAllowedTools` including `vault.expansion_pass`),
enable the tool in Nexus config, restart the Connector, then schedule a
Calendar event. The first real run is write-capable and consequential.
